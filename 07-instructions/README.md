# 07 — Instructions & Assembly

In module 06 we programmed the CPU by hand-encoding bytes like `0x4E` and
counting memory addresses to figure out where jumps should land. That's exactly
how the earliest programmers worked — and it's exactly why **assemblers** were
invented. This module builds one: a program that turns readable assembly text
into the machine code our CPU runs. It's the first rung on the ladder that leads
all the way up to high-level languages.

## The concept in plain language

**Assembly language** is a thin, human-friendly skin over machine code. Instead
of `0x4E` you write `ADD I`; instead of memorizing that the loop starts at
address 4, you write `JMP loop` and let the tool figure out the address. The
assembler's job is a faithful, mechanical translation:

- **Mnemonics** — `LOADI`, `ADD`, `JMP`, … map one-to-one to opcodes.
- **Operands** — a number (`5`, `0xFF`) or a **label**.
- **Labels** — names for addresses. `loop:` marks a spot; `JMP loop` refers to
  it. Labels work for code (jump targets) *and* data (variables/constants).
- **Comments** — everything after `;` is for humans and ignored.
- **`DB`** — a directive that places a raw data byte in memory (for constants
  like `-1` and variables like a running total).

## Why it matters

This is the **hardware/software boundary**. Below the assembler is physics —
gates, a control unit, the fetch/decode/execute loop. Above it is *language*: a
notation people can actually read and reason about. Every layer above (compilers,
interpreters, the Python running this repo) is more of the same idea — translate
a more expressive notation down toward the one beneath it. The assembler is
where that ladder starts.

The key technical idea is the **two-pass** design, which solves *forward
references* (`JMP loop` before `loop:` is defined):

1. **Pass 1** walks the source, assigns one memory word per instruction/`DB`,
   and records each label's address in a **symbol table**.
2. **Pass 2** walks again and emits bytes, now able to resolve any label —
   forward or backward — to its address.

## How the code demonstrates it

**[`assembler.py`](assembler.py)** implements the full pipeline: strip comments,
split off `label:` prefixes, run the two passes, and emit a list of byte values
ready to drop into the CPU's memory. It reports friendly errors with line
numbers for unknown instructions, undefined or duplicate labels, and operands
that don't fit the 4-bit field.

**[`programs/`](programs/)** holds real `.asm` source:

- [`sum.asm`](programs/sum.asm) — sums 1..5 with a `loop:`/`done:` structure and
  `DB` variables. The tests confirm it assembles to the **byte-for-byte same
  machine code** we hand-built in module 06.
- [`countdown.asm`](programs/countdown.asm) — prints 3, 2, 1 using `OUT` in a
  loop.

**[`demo.py`](demo.py)** assembles both, prints the generated bytes beside a
disassembly, and runs them on the module-06 CPU — closing the loop from *text →
machine code → execution*.

## Run it

```bash
# from the repo root

# assemble the example programs and run them on the CPU
python 07-instructions/demo.py

# run the tests
pytest 07-instructions/
```

## Files

- `assembler.py` — the two-pass assembler (`assemble`, `assemble_file`).
- `programs/sum.asm`, `programs/countdown.asm` — example source.
- `demo.py` — assemble, disassemble, and execute end to end.
- `test_assembler.py` — translation, labels (forward/backward), error cases, and
  end-to-end runs; verifies output matches module 06's hand-encoded bytes.
